# Settings Information-Architecture Audit

_Generated by the `settings-ia-audit` workflow (61 agents). Saved for a later redesign pass — not yet implemented._

## Current state

Settings are scattered across 6 disconnected surfaces with no organizing logic and no single source of truth. The hub is ProfileSettingsModal ("SYSTEM SETTINGS", ProfileSettingsModal.tsx:419) with 4 tabs whose array is literally ['profile','appearance','integrations','advanced'] (line 423) - named for storage buckets, not user goals. The Integrations tab (lines 762-1507) is a single 1500-line vertical scroll of 13 fully-expanded cards mixing three unrelated domains (AI media providers, third-party services, the user's own data stores) with no headers, anchors, or search, using three different on/off idioms. The two tallest cards (Image Engine + Image Understanding) lead the tab and push commonly-used controls below the fold. Around this hub orbit 5 surfaces that re-own the same settings: OnboardingWizard (8 steps, its back half - relay/Tailscale/iOS Shortcut - is dense developer setup), ModelWizardModal (a divergent provider list), VoiceSetupModal (the real voice picker, in component-local state so it cannot be deep-linked), MessagesSetupWizard (a duplicate Full Disk Access flow with inverted setupComplete open logic), and AssistantSettingsModal (per-agent overrides on a different, hover/right-click-only gear). Verified: TTS voice has 3 owners with pitch existing ONLY per-agent (AssistantSettingsModal.tsx:224 has pitch; VoiceSetupModal and Appearance do not); the per-agent voice already inherits the app default (AssistantSettingsModal.tsx:220-221); provider API keys are entered in 6+ places and the 'Inheriting from Chat Models' badge HIDES the editable field (ProfileSettingsModal.tsx:787-795); the Default Model dropdown shows raw IDs and an opaque 'Use current model' default (AssistantSettingsModal.tsx:163-165); labels leak jargon ('About Me (Global Context)' line 476, 'Allow Profile Updates' line 482, GUS, IMAP, Bot Token, Full Disk Access, Context Window Line, Local Planner). No tab/step state persists and no search exists, so a returning user cannot land on a known control in under 10 seconds.

## Core problems

- **Organized by data-type/internal architecture, not by what the user is trying to do. Tabs are named for storage buckets so any goal-driven user ('make web search work', 'change the reading voice', 'connect my phone') has no obvious destination and must scan everything.** — ProfileSettingsModal.tsx:423 - tab array is literally ['profile','appearance','integrations','advanced']; the audit's logical groupings exist only in the inventory, never rendered. Integrations is one unsegmented scroll (lines 762-1507).
- **No single source of truth: each concept has 2-3 owners with no cross-references, producing the 'is THIS the real setting?' doubt. This is the deepest cross-cutting failure the synthesis must resolve completely.** — TTS voice: 3 owners (Appearance launcher 748-760, VoiceSetupModal real picker, AssistantSettingsModal per-agent 218-226) and pitch only per-agent (224). API keys: ModelWizard + Onboarding + image card (797-814) + vision (925-941) + Brave (1010) + Tavily (1029) + Slack (1052) + GUS (1458), with the inherit badge hiding the field (787-795). iMessage FDA flow duplicated in MessagesSetupWizard.tsx:25/123 and ProfileSettingsModal.tsx:1194-1242. Name/bio and Brave co-owned by Onboarding and Settings.
- **No progressive disclosure: the rarest, heaviest config leads and complexity is never hidden, so non-technical users freeze and power users scroll past what they don't need.** — ProfileSettingsModal.tsx:765-991 - Image Engine + Image Understanding (each = provider+key+endpoint+fetch+model+test) render first, fully expanded, pushing web search, Slack, iMessage, Mail below the fold.
- **Recommendation-last instead of recommendation-first. The make-or-break first-run model step drops a non-technical user onto a raw API-key password field with a developer-console link and no plain explanation it is free/safe; web search shows two unfamiliar brands as equals.** — OnboardingWizard.tsx StepModel/ModelConnectForm 412-433 (blocker: API key with no plain-language explanation); Brave card 994-1013 followed by Tavily 1016-1032. Conflicts with owner's recommendation-first, one-click, Apple-like preference (Model Setup UX memory).
- **Pervasive engineer jargon in the labels users scan first, with no reassurance next to alarming permissions.** — 'GUS' (1436), 'IMAP' (1081/1101), 'Bot Token'/'xoxb-' (1041), 'Full Disk Access' unexplained (1223), 'Bearer Token' (Onboarding 1253), 'Access Token / Session ID' + CLI-only (1458/1468), 'Context Window Line' (1484), 'Local Planner' (1499), 'Target/Vision Model ID' (834/958), 'Canvas Artifact' (891), 'About Me (Global Context)' (476), 'Allow Profile Updates' (482), 'Core Drive' (Assistant 185), 'Run Dream Cycle' (Assistant 335), 'Polish' (Assistant 151), 'File Access Grants' (1525).
- **Heavy, optional developer setup (capture relay, Tailscale, hand-built iOS Shortcut with bearer token) is embedded mid-flow in the mandatory-feeling first-run wizard, making onboarding feel impossible.** — OnboardingWizard.tsx StepRelay (885+), StepTailscale (1035-1173), StepShortcut (1177-1304) occupy steps 6-8 of 8 with 'local web server', 'Tailscale', 'bearer token', 'instanceId', 'shareId', JSON request body.
- **Inconsistent control idioms and inconsistent modal state ownership break learnability and deep-linking, and per-agent settings are on a near-identical gear users can't find.** — Three on/off idioms in one tab - Enable pill (Brave 1006, Tavily 1025, Slack 1044, GUS 1441) vs segmented backend selector (Calendar 1256-1264) vs provider dropdown (Image 773, Vision 909). VoiceSetupModal & MessagesSetupWizard use local state and aren't deep-linkable; MessagesSetupWizard uses inverted setupComplete logic; profileSettingsTab/assistantSettingsTab reset on mount. AssistantSettingsModal reached only via ChatHeader.tsx:225 or AppSidebar right-click 236.

## Proposed information architecture

Adopt the by-task hub (Proposal 1) as the skeleton because users navigate by goal, but graft in the two structural insights it underweights: (a) the explicit SCOPE boundary from Proposal 2 - App-level vs This-Agent vs First-Run, made visible so users never confuse 'app default' with 'this agent's override'; and (b) Proposal 3's clean split between AI MODELS (the brains + one key vault) and YOUR DATA (personal stores) so the 13-card wall becomes legible. The unified hub is 'Settings' (drop 'SYSTEM') with a persistent search/filter box on top and 5 task sections, each opening recommendation-first with complexity behind 'Advanced' disclosures, and each deep-linkable from a single store-backed open API. Sections: (1) YOU - identity, voice, look (Voice MOVES out of Appearance because read-aloud is a personal-output concern, not theme; pitch added at app level to make the override symmetric); (2) AI MODELS - every brain the app thinks with plus the single canonical API-key vault every feature references read-only; (3) CONNECT YOUR APPS - third-party accounts you sign into AND your personal data stores, one consistent enable-then-expand card pattern, frequent cards first, optional iPhone capture moved here out of first-run; (4) PRIVACY & CONTROL - developer mode, folder permissions, activity log; (5) a persistent link-out to the agent's own settings. The 6 surfaces collapse to ONE settings store: VoiceSetupModal and MessagesSetupWizard flows become shared components embedded in their canonical sections; the wizards become guided first-passes that WRITE THROUGH to these sections and say 'change this later in Settings' rather than re-owning state.

*Role:* Entry hub titled 'Settings' (drop the shouty 'SYSTEM SETTINGS'). Opens to a short scannable list of the 5 sections with a persistent filter/search box that jumps to any control by name. Remembers last-visited section (fixes the reset-on-mount bug). All open/section state lives in one store behind a single open API so any control is deep-linkable.  
*Reached from:* AppSidebar gear (make it always-visible, not hover-only), StartPage SYSTEM SETTINGS button, command palette; each section deep-linkable via the settings store

### What merges

- **TTS read-aloud voice - Appearance launcher (ProfileSettingsModal.tsx:748-760), the real picker (VoiceSetupModal), and per-agent override (AssistantSettingsModal.tsx:218-226)** → ONE canonical 'You > Reading voice' panel that IS the picker (voice + speed + pitch inline), promoted into the settings store so it is deep-linkable. Per-agent voice stays as an explicit override labeled 'Overrides app default voice' showing the resolved fallback in place. — Voice has 3 owners and pitch exists ONLY per-agent (verified AssistantSettingsModal.tsx:224), so a user changing the default pitch hunts and concludes it's unavailable. The per-agent picker already inherits the app default (verified 220-221), so the only missing piece for a symmetric fallback chain is adding pitch at app level and making the app panel the single editable home. Moving it out of Appearance fixes the mis-filing (voice is output, not theme).
- **Every provider API key - ModelWizardModal, OnboardingWizard, Image Engine card (797-814), Image Understanding card (925-941), Brave (1010), Tavily (1029), Slack (1052), Mail, GUS (1458)** → A single 'AI Models > API Keys' vault listing each provider key exactly once as the source of truth. Image/vision/feature cards REFERENCE it read-only ('Using your Google key - manage in API Keys') with an explicit per-feature override. The inherit badge keeps the status but links to the canonical field instead of hiding all editing. — Keys are scattered across 6+ surfaces and the 'Inheriting from Chat Models' badge hides the editable field (verified 787-795), so there is no authoritative place to rotate a key. One vault + explicit, consistent, visible inheritance is the single highest-leverage fix for source-of-truth.
- **iMessage Full Disk Access grant+verify, duplicated in MessagesSetupWizard.tsx:25/123 and ProfileSettingsModal.tsx:1194-1242** → ONE shared grant+verify component owned by 'Connect your apps > Messages', reused by the first-run wizard. The wizard explicitly names Settings > Connect your apps > Messages as the permanent reconnection home. — Two independent copies of the same macOS permission flow drift, and the wizard never re-shows after setupComplete (verified inverted-logic flag), silently relocating reconnection. A shared component + a wizard that points at the settings home removes both the duplication and the orphaning.
- **ModelWizardModal connect flow + Image Engine card + Image Understanding card, and the divergent OnboardingWizard PROVIDERS array vs ModelWizard provider dropdown** → AI Models section, with ONE shared provider catalog (ids, display names, endpoints, local-detection) used by the first-run model step and the 'Add a model' flow; deep-linkable rather than re-running the full 8-step wizard. — All three configure models the AI thinks with, yet chat models live in a separate modal while image/vision sit in Integrations and invisibly inherit chat keys; connectable providers even differ by which door you enter. Co-locating them and sharing one catalog makes 'the brains' one place and fixes the 'Re-run Setup Wizard funnels a model fix into 8 steps' anti-pattern.
- **The Brave card and Tavily card shown side-by-side as equals (994-1013 + 1016-1032)** → ONE 'Web search' toggle pre-set to the recommended default (Brave) on, with Tavily behind 'Use a different provider' and a 'Pick one - you don't need both' line. — Two unfamiliar brands with no guidance causes decision paralysis on what the user assumes is a single on/off switch (verified). Recommendation-first collapses it to one switch.
- **The 13 unsegmented Integrations cards using 3 different on/off idioms** → Split across AI Models (image/vision/keys) and Connect your apps (everything external), with ONE consistent card pattern: header toggle + collapsed body that expands to configure. Reserve the segmented selector ONLY for the genuine Local-vs-Apple backend choice. — Three idioms and a wall of expanded cards force relearning and scrolling past irrelevant cards. One pattern + collapse-by-default + a domain split delivers consistency and progressive disclosure.
- **Onboarding StepProfile name/bio (164/166) and StepBraveSearch (763-796) which re-own settings that also live in Settings** → Settings is canonical ('You' for name/bio, 'Connect your apps > Web search' for Brave); Onboarding writes the same keys but each step carries 'You can change this later in Settings.' — Name/bio and Brave are co-owned with neither surface signaling the other. Treating Settings as canonical and the wizard as a deferring first-pass keeps one home without removing the value of guided first-run.

### What stays separate

- **AssistantSettingsModal (per-agent: system prompt, role, core drive, default model, per-agent voice, tool enable + account restrictions, agent memory/docs/pins)** — These are scoped to ONE agent, not the app; merging them into the global hub would conflate 'app default' with 'this agent's override' and bloat the hub. Keep it separate but (a) reach it from an always-visible control on the agent (not hover/right-click-only, verified ChatHeader.tsx:225 / AppSidebar.tsx:236), (b) add the hub link-out from AI Models, (c) label every override explicitly ('Overrides app default voice') with the resolved fallback shown in place, and (d) give friendly model labels + 'Same as the rest of the app (default)' instead of raw IDs and 'Use current model' (verified AssistantSettingsModal.tsx:163-165).
- **OnboardingWizard (first-run guided pass)** — A guided first pass is legitimately different from a reference settings surface - it teaches, sequences, and writes through. Keep it, but slim it to the 3 promised essentials (profile, a model, web search), make it recommendation-first, have every step say 'change this later in Settings', deep-link into the canonical sections, and move the relay/Tailscale/iOS-Shortcut back half OUT into the optional 'Set up iPhone capture' card.
- **MessagesSetupWizard (first-open iMessage walkthrough)** — A guided first grant of macOS permissions on first Messages open has real onboarding value, so it stays as a launcher - but it MUST use the SHARED grant+verify component and name 'Connect your apps > Messages' as the permanent home, rather than carrying its own copy of the flow with inverted open logic.
- **Local-vs-Apple backend selector for Calendar / Reminders / Notes** — This is a genuine either/or backend choice, not an on/off, so the segmented selector idiom is appropriate here and ONLY here - it stays distinct from the standard header-toggle card pattern.
- **Theme + Accent color (kept, but folded into 'You' rather than a standalone Appearance tab)** — Pure look-and-feel is small and is a YOU-domain concern; once Voice leaves Appearance, two color controls don't warrant a top-level tab. Folding them into You keeps the top-level list to 5 scannable sections without losing the controls.

## Dummy-proofing / clarity fixes

- **Default-land on the 'You' section (zero jargon: name, About you, voice, theme) and quarantine all technical surfaces in 'Privacy & control'. Rename the hub from 'SYSTEM SETTINGS' to 'Settings'.** — A non-technical user opens to friendly identity/voice/look settings and is never confronted first by API keys, Full Disk Access, or relays. 'SYSTEM SETTINGS' in all-caps reads as developer territory (verified ProfileSettingsModal.tsx:419).
- **Rename every section and card to the job, not the tech: sections 'You / AI Models / Connect your apps / Privacy & control'; cards 'Email', 'Messages', 'Web search', 'Salesforce Work Items (GUS)', 'Show the memory line in chat', 'Built-in task list', 'Reading voice', 'Folders agents can use', 'About you', 'Let agents suggest profile edits - you approve each change', 'Which model to use', 'Where new images appear: Open in Canvas / Show in chat'. Per-agent: 'Guiding principle' (Core Drive), 'Review & organize memories' (Run Dream Cycle), 'Improve with AI' (Polish).** — Users navigate by goal; data-type/architecture names force a translation the non-technical persona can't do. Each rename is a specific verified plain-language finding (lines 476, 482, 834, 891, 1436, 1484, 1499, 1525; AssistantSettingsModal 151, 185, 335).
- **Recommendation-first defaults everywhere: Web search is a single toggle pre-set to Brave with Tavily behind 'Use a different provider'; AI Models leads with the recommended model and one-click add; image/vision endpoints hidden until 'Advanced'; the first-run model step leads with a curated Local-vs-Cloud recommendation, not a raw key field.** — Directly answers the decision-paralysis and recommendation-last blockers (two web-search brands as equals; API key field with no default path) and matches the owner's stated 'recommendation-first, one-click, Apple-like' preference (Model Setup UX memory).
- **Pair every credential field with (1) a plain 'it's free & safe' sentence, (2) a guided walkthrough of the provider signup (reuse the Mail app-password step pattern at ProfileSettingsModal.tsx:1121-1148), and (3) a Test arrow 'Connected!' confirmation. Disabled Test buttons show an inline reason ('Enter a model to test') instead of dead-clicking.** — Bare key fields with cryptic placeholders ('AIza...','BSA...','xoxb-','sk-') and silently-disabled Test buttons (verified 858-865, 972-979) are the single biggest recurring wall across all personas.
- **Add reassurance next to alarming permissions: by 'Open Full Disk Access' add 'macOS keeps your messages locked behind a setting called Full Disk Access. - **Progressive disclosure as the default layout: every service card is collapsed to a one-line summary + header toggle and expands on demand; the heavy Image/Vision/advanced-endpoint config and per-person voices live behind 'Advanced' / 'Show advanced steps'; frequently-toggled low-height cards lead. One consistent enable+configure idiom (header toggle + expanding body) across all on/off integrations; segmented selector reserved for the Local-vs-Apple backend choice only.** — Stops the rarest, tallest cards (image config) from pushing common controls below the fold (verified 765-991) and ends the three-idioms-in-one-tab relearning cost.
- **Single source of truth made visible: each API key and the reading voice live in exactly one place; feature and per-agent surfaces show 'Using your Google key - manage in API Keys' / 'Overrides app default voice' with the resolved value, the inherit badge links to the canonical field instead of hiding it, and pitch is added at app level so the override is symmetric.** — Removes the 'is THIS the real setting?' doubt, the hidden-field problem (787-795), and the asymmetric pitch gap (224) - the deepest cross-cutting failure.
- **Make all settings surfaces store-backed and deep-linkable behind one open API: promote VoiceSetupModal and MessagesSetupWizard out of local state, replace the inverted setupComplete open logic, persist last-visited section, add a top search/filter box, make the sidebar gear always-visible, and add an always-visible per-agent settings control plus a hub link-out.** — Local-state modals can't be deep-linked, the hover-only gear is undiscoverable, tabs reset on mount, and users open the wrong of two near-identical gears (verified stateMap inconsistencies and deepLinkable entries) - all of which block the returning user's under-10-second target.

## Verified UX findings

- **[major] Step 3 demands an API key with no plain-language explanation of what it is or whether it costs money** (OnboardingWizard.tsx StepModel / ModelConnectForm (lines 412-433); 'Google AI API key', placeholder 'AIza…', 'Get key' link) — A non-technical first-time user has never heard of an 'API key.' The wizard jumps from a friendly two-box recommendation straight to a password field labeled 'Google AI API key' and a link that throws them onto Google's developer site to create something they don't understand. There is no inline explanation of what a key is, that it's free, or that it's safe. This is the make-or-break moment of first-run and it's pure developer jargon. Many parents will either bail here or push through anxiously, unsure if they just signed up for charges.  
  → _Before the key field, add one plain sentence: 'An API key is a free password from Google that lets this app talk to their AI — it usually takes 1 minute and costs nothing for normal use.' Better: walk the user through the Get-Key page with a screenshot or numbered steps (you already do exactly this for Mail app passwords at lines 1121-1148 — reuse that pattern here). Consider a true one-click/OAuth path so a key never has to be hand-copied._
- **[major] Connecting personal email requires generating an 'app password' and understanding IMAP, not a normal login** (ProfileSettingsModal.tsx Integrations tab, Mail accounts card (lines 1073-1148); 'Gmail & iCloud over IMAP — app password, no web login') — The user's mental model of 'connect my email' is 'type my email and password.' Instead they hit the unexplained term 'IMAP,' a confusing 'no web login' phrase, and a multi-step detour to Google's app-passwords page that warns them to enable 2-Step Verification first and that they'll 'only see it once.' This is far heavier than a non-technical user expects and a very likely give-up point. The error-handling hint at lines 1182-1187 even anticipates users pasting their normal password — confirming this trips people up.  
  → _Lead with reassurance and hide the jargon: rename the card to plainly 'Email' and drop 'IMAP' from the headline. Add a one-line 'Why a special password? Google/Apple require it for security — we'll show you exactly how' and keep the existing good step-by-step. Long term, offer Gmail/iCloud OAuth (a real in-app sign-in) so most users never touch an app password._
- **[major] Onboarding back-half (Relay, Tailscale, iOS Shortcut, bearer token) is dense developer jargon mid-wizard** (OnboardingWizard.tsx StepRelay (885+), StepTailscale (1035-1173), StepShortcut (1177-1304); terms 'local web server', 'Tailscale', 'bearer token', 'instanceId', 'shareId') — Steps 6-8 of an 8-step first-run flow ask a non-technical user to install a 'capture relay' local web server, download and configure Tailscale on both Mac and iPhone, and hand-build an iOS Shortcut with a bearer token and a JSON request body. # Post-review corrections (2026-06-23)

_These corrections were made after pressure-testing the audit's recommendations against the actual code (a step the original audit skipped — it verified the **problems** but never tested whether the **fixes** were feasible). **Where anything here conflicts with the recommendations above, this section wins.**_

## Verified mistakes in the recommendations above

1. **Email / "make it a normal login" + OAuth** — wrong. Email is IMAP-by-design; the app-specific password is **mandatory** (no password-less path with 2FA on). OAuth is a heavy, unrealistic build for a local app. → Keep the app-password step; make it clear & reassuring ("Email" card, drop "IMAP" from the headline, "why a special password — Google/Apple require it, here's how"). Do **not** pursue email OAuth.

2. **"Brave search on by default / free & safe"** — wrong. Brave needs a user `BSA…` key; enabling it without one **fails silently** (`webSearch.ts:65` requires `enabled && apiKey`; no warning, unlike Tavily). → Never ship Brave pre-on. AND see the browser-search finding below — web search does **not** require a key at all.

3. **"Add 'free & safe' to every credential"** — wrong. OpenAI/Anthropic keys cost money; Salesforce needs a license. → Honest per-credential labels only. Local = genuinely free & private. Recommend **Gemini** (real free tier) as the friendly default and direct users to where they get the key; never label OpenAI/Anthropic "free."

4. **Single "API-key vault / canonical source of truth"** — over-engineered and conflicts with the data model. Keys are stored **per-model** (`models[].apiKey`); two same-provider models can hold different keys; image/vision today grab the *first* match via `.find()`. → Drop the vault concept. What's actually wanted: **clean API-key inputs per provider** + a reliable, accurate **"Fetch models"** button. Pricing in that list is **not feasible** (provider APIs don't return pricing) — skip it or use a static "see pricing" link.

5. **"Link the inherit badge to the canonical field"** — impossible as stated; there is no single canonical field (inherited key lives in some `models[i].apiKey`, separate from `integrations.<provider>.apiKey`). → Folds into #4; show which model a key comes from, or an explicit override toggle.

6. **Model OAuth (Phase 5)** — infeasible. Google/OpenAI/Anthropic are API-key-only. → Dropped. Invest in making the key flow smoother (validation, masked keys, guided signup pages).

7. **"Recommend a Local model in first-run"** — keep it, but it is **not** zero-config (5–20 GB model download + a local inference engine). → Build a proper **guided setup screen**: show the download + config steps, and while it works, tell the user plainly "this takes a few minutes but runs 100% private on your Mac." Local stays a real, valid recommendation alongside Google's free tier.

8. **Rename "Context Window Line" → "Show the memory line in chat"** — wrong; "memory line" collides with the app's actual Memory panel. → Use "Show where context ends in chat" / "Context window divider."

## New finding — keyless agent-controlled browser search ALREADY EXISTS

Verified, production-ready, and the adversarial re-check confirmed it. The `browse` capability (`src/services/capabilities/builtins/browse.ts`, `browserAgent.ts`, `browserAnnotator.ts`, Tauri commands in `src-tauri/src/lib.rs`) opens a real browser tab, defaults to the **keyless** DuckDuckGo HTML endpoint, and runs an observe→decide→act loop (navigate/read/click/type/scroll, ~12 steps) with safety guards (auth blocklist, injection delimiters, password fields hidden, submit confirmation). The embedded browser is fully agent-drivable.

- **Implication:** web search is **not** gated on an API key. Brave/Tavily are an *optional* "faster, cleaner results" upgrade, not a requirement. Reframe the whole web-search section around this.
- **Only gap (routing, ~half-day):** plain "search for X" prefers the API path when a key is set; the browser path fires on "browse/open/navigate" phrasing or when forced. Extend `routeToolCandidates` in `memoryGatekeeper.ts` to fall back to the browser when a search is intended but no Brave/Tavily key is configured. Do **not** rebuild search from scratch.

## "Right idea — refine the how, not the whether"

- **Shared provider catalog** — do it, but extract a shared spec both adapt to; the two catalogs use different shapes + IDs (`gemini` vs `google`) and onboarding has a local option. Don't naively merge.
- **App-level voice pitch** — the field already exists in the data model; it's just not wired to the picker UI/preview. "Finish wiring it," not "add it."
- **Promote modal state to the store** — only the opened/step state; leave transient, browser-coupled state (voice list, playback) local.
- **Shared iMessage FDA component** — the two flows persist differently (wizard doesn't write `imessage.enabled`, settings does); share via a callback param, not one rigid component.
